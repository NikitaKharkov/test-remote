<!--                                          INIT RESPONSE  
              This node will be returned as a result of a "INIT" request.  The 
              purpose of the INIT request to establish a session for a particular
              user of the caller's site.  The key element in this request
              will be the SessionID attribute of the root node.  This should be
              used on all subsequent request for this user for the current session.  
              (Note that, if there is more than 10 minutes of inactivity, the current
              session may be dropped at EBSCO,  if a request come in after that
              time, a new session will be created and a new SessionID assigned.  It
              is important to track the latest sessionID for the user.)
              
              The caller will be returned the following nodes as part of this response:
                        errors:  A list of errors specific to the login.  Could include
                                    things like bad user ID, profile not found, etc.
                        dblist:  The list of databases the user can search. For each 
                                    database will be the long name (if the caller wants 
                                    to display) and the code for use in SEARCH requests.
                        userLimits: lists various restrictions for the user, such 
                                    as maximum number of hits, maximum number of results
                                    that can be viewed for a given search.
                        otherInfo: not used, but could be in the future.
-->
<!ELEMENT initResponse (errors?, dbList, userLimits?)>
<!ELEMENT dbList (db+)>
<!ELEMENT db (dbCode | dbName | dbType | dbSortOptions?)*>
<!ATTLIST db
            seqNo CDATA #IMPLIED
>

<!ELEMENT MaxResultSetSize (#PCDATA)>

<!--                                          SEARCH RESPONSE        
              This node will be returned as a result of a "SEARCH" request.  
              
              The resultSetID attribute of the searchResponse should be retained by the
              caller for use on subsequent retrievals (Present Requests) for the given
              search.
              
              The caller will be provide with the following nodes (if applicable):
                        
                        Errors-  lists any overall errors encountered with the protocol
                                    (any specific search errors will be shown under the 
                                    searchResults node and any retrieval errors will be shown 
                                    under the individual records.)
                        
                        RequestInfo - Restating the request and how it was applied,
                                    e.g. the query used, the databases, number or records
                                    requested, format requested, any limits and expanders
                                    applied (defaults used if not added to the actual request)
                        
                        SearchResults - Basically this holds the total hits.
                        
                        Records - this section stores the requested records resulting
                                    from the search.
-->

<!--                                          PRESENT RESPONSE      
              This node will be returned as a result of a "PRESENT" request.  
              The response is identical to the Search Response.  This is done
              on purpose to allow the caller to get all of the search information
              without having to store this state on their end.   
              
              The PresentResponse returns the ResultSetID for ease of reference.  This
              should be identical to the one submitted by the caller in the request
              except if the request "timed-out" and the search was re-exectuted.  The
              caller should treat this as the correct resultSetID for subsequent
              Present Requests.
              
              The caller will be provide with the following nodes (if applicable):
                        Errors-  lists any overall errors encountered with the request.
                        RequestInfo - Restating the request and how it was applied,
                                    e.g. the query used, the databases, number or records
                                    requested, format requested, any limits and expanders
                                    applied (defaults used if not added to the actual request)
                        SearchResults - Basically this holds the total hits.
                        Records - this section stores the the requested records resulting
                                    from the search.
-->
<!ELEMENT presentResponse (requestInfo?, searchResults, records?)>
<!ATTLIST presentResponse
            resultSetID ID #IMPLIED
> 
<!-- ============================================================================== -->
<!--                                              TERMINATE RESPONSE           
              This node will be returned as a result of a "TERMINATE" request.  The 
              reponse is very simple and merely provides a status of OK or Failure.  
              If failure, then the "errors" node should describe what went wrong.
              
              This is informational only, the caller is not expected to take any
              action on a "failed" request for termination.
-->
<!ELEMENT terminateResponse (errors?)>
<!ATTLIST terminateResponse
            status (ok | failure) #REQUIRED
>

<!-- inProfile, Y/N indicator if parent/target article is on the user's profile (User Specific). Added 3/2009 for JIRA EHST-3070 / NetCRM SI 21012. Problem: The parent/target article of the IQV image could be in the user’s group but not in the user’s profile. When viewing the image detail, and clicking on the 'Original Article' link, the user will get redirected to the search screen with a 'No Results Found' message. Fix: Now, CE will add an 'inProfile' attribute to rightsinfo. Method: CE will first make an AMS call with just the DBs in the user’s profile and try to get an AMS match. If that succeeds, it sets rightsinfo db and ui to the appropriate target DB/UI and sets inProfile to 'Y'.If it fails, CE will set 'inProfile' to 'N' (must include the attribute). CE will then make a second AMS call using the group databases. If it finds a match, it will set rightsinfo db and ui and leave inProfile to 'N'. Outcome: Ehost UI uses rightsinfo db and ui to make a plink, which it will do regardless of whether or not the DB is in the user’s profile. However, it only displays the 'Original Article' link if inProfile='Y'. -->
<!ELEMENT rightsinfo EMPTY>
<!ATTLIST rightsinfo
            db CDATA #IMPLIED
            ui CDATA #IMPLIED
            inProfile (Y | N) #IMPLIED
>
