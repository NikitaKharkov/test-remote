Enterprise Search Integration Overview:

Enterprise Search Integration (ESI) makes database metadata available for local indexing. Databases are accessible as XML via FTP.

XML files contain article metadata with links back to Full Text. XML files of the article metadata are put onto an ftp site as content is updated inside EBSCOhost.

Customer setup tasks

  • Customer Enterprise Search Application administrator scripts pick-up and process the XML.
  • Perform initial full database load.
  • Retrieve daily/frequent updates.

Enterprise Search Integration FTP Retrieval:

The FTP directory for your provided username and password are organized by folders named by the product codes.

 

The manifest.txt file in each directory lists the order to download and index the files, where each file is separated by a line break. The first file to download and index is on line one, the second file on line two, the third on line three and so forth. The manifest.txt file is erased and recreated when a new full build becomes available.

Examples:

·         heh1_20080623_full-001.zip

·         heh1_20080623_full-002.zip

·         heh1_20080623_full-003.zip

·         heh1_20080623_full-004.zip

·         heh1_20080623_full-005.zip

·         heh2_20080623_full-001.zip

·         heh2_20080701_update-001.zip

·         heh2_20080706_update-001.zip

·         heh2_20080713_update-001.zip

·         heh2_20080721_update-001.zip

 

[Square brackets indicates optional elements]

 

ZIP files are named in each folder using the following syntax: {productcode}[{segment}]_{yyyymmdd}_full-{counter}.zip

productcode

short database code

[segment]

database segment

yyyymmdd

date of file

counter

sequential counter

 

XML files are named in the ZIP files using the following syntax: {product code}[{segment}]_{yyyymmdd}_update-{counter}.xml

 

productcode

short database code

[segment]

database segment

yyyymmdd

date of file

counter

sequential counter

Enterprise Search Integration FTP Retrieval:

XML database rebuild files represent metadata in the following format:

<root>
<record plink="[Persistent link to the EBSCOhost article] ">
<item name="[Metadata field]">[Field value]</item>
<item name="[Metadata field]">
<subitem name="[Metadata subfield]">[Subfield value]</subitem>
<subitem name="[Metadata subfield]">[Subfield value]</subitem>
</item>
<item name="[Metadata field]">[Field value]</item>
...
</record>
<record plink="[Persistent link to the EBSCOhost article] ">
<item name="[Metadata field]">
<subitem name="[Metadata subfield]">[Subfield value]</subitem>
<subitem name="[Metadata subfield]">[Subfield value]</subitem>
</item>
...
</record>
...
</root>

 

Three types of records are represented in XML update files:

 

1.     New - The metadata for the record is the same as in the full build. The record needs to be added to the database index.

2.     Update - The metadata for the record is the same as in the full build. The entire record needs to be reloaded with the new data.

3.     Delete - These records will have the delete attribute set to true in the record xml element, and the only subelement of this record will be an item xml element containing the AN for the record. An example would look like this:

 

<record plink="http://search.ebscohost.com/login.aspx?
direct=true&id=113603&site=dynamed" delete="true">
<item name="AN">113603</item>
</record>
